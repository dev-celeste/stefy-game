Can you make pixel art in pure CSS? Yes — and here are three ways.

You asked whether pixel art can live in pure HTML/CSS. It absolutely can. This page builds the same Stefan sprite three different ways, cheapest trick to cleverest — and the third one is exactly what your game now uses.

Pixel art is just a tiny grid of colored squares. So any technique that lets CSS draw a grid of colored squares — and keeps them crisp when scaled up — counts. The three below go from "one line of CSS" to "one element pretending to be a whole picture."

1. The pragmatic one: image-rendering: pixelated

1 line of CSS

This isn't drawing pixels in CSS so much as telling the browser "don't blur my pixels when you scale them." Take a small image (or SVG, or canvas) and blow it up. Normally the browser smooths it into mush. One property stops that:

img { image-rendering: pixelated; }
default (blurry when scaled)
image-rendering: pixelated (crisp)

Same 8×8 image, scaled 16×. This is the workhorse — it's why your game already looks crisp, and it's the property that makes the other two techniques below stay sharp too. If you only remember one thing, remember this line.

2. The literal one: a grid of <div>s

most intuitive

The most honest approach: make one <div> per pixel and color each with background. A CSS grid lays them out in rows and columns. It's a lot of elements (192 for Stefan), but conceptually it's dead simple — you're literally building the picture square by square.

Stefan, built from <div> pixels

Great for learning and for small sprites. The downside is element count: a full scene of these can get heavy for the browser. Which leads to the clever trick…

3. The clever one: one element + box-shadow

what your game uses

Here's the magician's trick. A box-shadow can be offset to any position — and you can stack as many shadows as you want on a single element, comma-separated. So take one tiny 8×8 dot, and give it one shadow per pixel, each offset to its spot and tinted its color. One element becomes an entire picture.

.dot {
  width: 8px; height: 8px;      /* one pixel */
  background: transparent;       /* the dot itself stays invisible */
  box-shadow:
     16px 24px 0 0 #e8c86a,      /* a blonde pixel at (col 2, row 3) */
     24px 24px 0 0 #e8c86a,      /* …another beside it… */
     /* …one line per lit pixel — your game generates these from an ASCII map */ ;
}

the actual element (the 1 real dot) is circled
Pixel size: 8px — drag it: the sprite rescales and stays perfectly crisp.

This is the exact technique now living in stefan.html. Your spriteShadow() function walks the ASCII pixel map and builds that giant comma-separated string — one shadow per lit character, skipping the transparent . cells. The whole cast is five tiny dots wearing a few hundred shadows between them.

So… which should you use?

For your game: the box-shadow trick (#3) — it kept everything in one self-contained file with zero image assets, which was the whole portability goal. And it doubles as a genuinely cool thing to have hand-rolled.

In general: reach for image-rendering: pixelated on a real PNG/SVG (#1) the moment you have actual art files — it's the least code and scales to detailed images that would be absurd as box-shadows. The grid-of-divs (#2) is best kept for teaching and truly tiny icons. Box-shadow (#3) shines exactly where you used it: small sprites, no external files.

A rule of thumb: hand-code the pixels (2 or 3) while art is placeholder and you want zero dependencies; switch to real image files (1) once you're drawing in Procreate. That's precisely the path the Art Swap Guide lays out.
